LLMs Are Incredibly Powerful — But…

An LLM is like a genius who read everything ever written — but was locked in a room at a fixed point in time, with no access to anything new.
  • ✓Can write code, reason, summarise, translate, plan — at expert level
  • ✓Trained on terabytes: Wikipedia, books, code, Stack Overflow, the web…
  • ✗Zero knowledge of anything after the training cutoff date
  • ✗Zero access to your company's private documents — ever
  • ✗When it doesn't know, it invents — confidently and convincingly

// four failure modes

4 Problems, 1 Solution

🌀

Problem 1 — Hallucination

LLMs fabricate facts when they don't know the answer. They present invented information with complete confidence — hard to detect, dangerous in production.

✓ RAG: answer is grounded in retrieved source text. Nothing invented.

📅

Problem 2 — Stale Knowledge

GPT-4o cannot answer about events after its training cutoff. New products, recent policy changes, this week's news — all invisible to it.

✓ RAG: update your document store — answers update instantly, no retraining.

🔒

Problem 3 — No Private Data

Your internal docs, customer records, proprietary research — none of it was in the training data. You can't ask GPT about your own company's specifics.

✓ RAG: ingest any private data into a local vector store.

💸

Problem 4 — Fine-tuning Cost

Retraining on custom data costs thousands of dollars, weeks of compute time, and ML expertise. Every data update requires another full training run.

✓ RAG: no training at all. Just update the documents.

What Is RAG?

Retrieval-Augmented Generation — give the LLM a cheat sheet of relevant facts right before it answers, so it never has to guess.

RAG has two distinct phases. Indexing happens once (or when docs change). Retrieval + Generation happens on every user query.

📥

Phase 1 — Indexing offline

Load documents → split into chunks → convert to embedding vectors → store in vector database. Runs once when setting up, or when documents are updated.

🔎

Phase 2 — Retrieval + Generation every query

User asks a question → embed the question → find similar chunk vectors → inject top chunks into the LLM prompt → LLM generates a grounded answer.

📄 Document Loader

Entry point of the pipeline. Reads raw data from any source and converts it into a standard Document object with page_content + metadata.

Available Loaders
PyPDFLoader
CSVLoader
WebBaseLoader
NotionLoader
SQLLoader
What you get back

A list of Document objects. Each has:
• page_content — the raw text
• metadata — source, page number, etc.

# Load a PDF — returns one Document per page from langchain_community.document_loaders \ import PyPDFLoader loader = PyPDFLoader("book.pdf") pages = loader.load() print(len(pages)) # → 250 (one Document per page) print(pages[0].page_content[:100]) # → "Chapter 1: Design Patterns..." print(pages[0].metadata) # → {"source": "book.pdf", "page": 0}
// component 2 · text splitter

✂️ Text Splitter

Splits large documents into smaller, overlapping chunks that fit the LLM context window and can be retrieved with precision.

Why overlap matters

Without overlap, a sentence spanning a chunk boundary would be split in half and lose meaning. Overlap ensures context is always preserved.

Sweet spot
Too Large
Noisy retrieval, hits context limits
Too Small
Loses context, misses answers

✓ 500–1500 chars, 10–20% overlap

from langchain_text_splitters import \ RecursiveCharacterTextSplitter # Tries separators in order — respects # natural text boundaries splitter = RecursiveCharacterTextSplitter( chunk_size = 1000, # max chars per chunk chunk_overlap = 200, # shared chars at edges separators = ["\n\n", "\n", ".", " "], ) chunks = splitter.split_documents(pages) print(len(chunks)) # 250 pages → ~800 chunks
// component 3 · embeddings

🧮 Embeddings

Converts text into a vector of numbers that captures semantic meaning. Texts with similar meaning produce vectors that are mathematically close together.

The key insight

"automobile" and "car" produce very similar vectors — even though the words are different. This is semantic similarity, not keyword matching.

Critical rule

You must use the exact same embedding model for:
• Encoding chunks during indexing
• Encoding the user query at runtime

Mixing models will give wrong results.

from langchain_openai import OpenAIEmbeddings embeddings = OpenAIEmbeddings( model="text-embedding-3-small" ) # Embed one query → list of floats vec = embeddings.embed_query("What is Singleton?") print(len(vec)) # → 1536 numbers print(vec[:3]) # → [0.023, -0.187, 0.412] # Texts with similar meaning → similar vectors # "car" ≈ "automobile" ≈ "vehicle" # → their vectors are close in 1536-D space
// component 4 · vector store

🗄️ Vector Store (ChromaDB)

A database purpose-built for storing and searching embedding vectors. It answers: "find me the K chunks most similar to this query vector."

How similarity search works

Computes cosine similarity between the query vector and every stored chunk vector. Returns the top-K chunks — most semantically relevant to the question.

Popular vector stores
ChromaDB — local free
Pinecone — cloud prod
FAISS — Meta fast
Weaviate — OSS
from langchain_community.vectorstores import Chroma # Store chunks + their embeddings vectorstore = Chroma.from_documents( documents = chunks, embedding = embeddings, collection_name = "book_rag", persist_directory= "./chroma_db", ) # Search: find 4 most relevant chunks results = vectorstore.similarity_search( "What is the Observer pattern?", k=4 ) for doc in results: print(doc.page_content[:80], "...")
// component 5 & 6 · retriever + prompt

🔍 Retriever & 📝 Prompt Builder

Retriever

A thin wrapper around the vector store. Takes a query string, runs similarity search, returns relevant Document objects. Can add score thresholds, re-ranking, or hybrid search.

retriever = vectorstore.as_retriever(search_kwargs={"k": 4})
Prompt structure
SYSTEM: Answer using ONLY the context. If unsure say "I don't know."

CONTEXT: {chunk 1 text}
———
{chunk 2 text}...

QUESTION: {user query}
ANSWER:
Why prompt structure matters
  • 1Instructing "ONLY the context" prevents the model from using its own training data
  • 2"Say I don't know" if not in context prevents hallucination
  • 3Context placed before the question gets better LLM attention
  • !Prompt quality directly determines answer quality

LangGraph — Graph-Based Workflows

LangGraph extends LangChain with a graph execution model — enabling stateful workflows with loops, branching, and multi-step agent reasoning.

// langgraph · 4 core concepts

LangGraph — State · Nodes · Edges · Compile

📦

State

A TypedDict shared across all nodes. Each node reads from and writes to it.

⚙️

Nodes

Plain Python functions. Input = current state. Return = dict of updates.

➡️

Edges

Fixed A→B or conditional if X: B routing between nodes.

🔒

Compile

Call .compile() → run with .invoke()

END

rag_graph.py def ingest_pdf(pdf_path: str) -> Chroma: # Step 1 — Load PDF pages = PyPDFLoader(pdf_path).load() print(f"Loaded {len(pages)} pages") # Step 2 — Split into overlapping chunks chunks = RecursiveCharacterTextSplitter( chunk_size=1000, chunk_overlap=200 ).split_documents(pages) print(f"Created {len(chunks)} chunks") # Step 3 — Embed + store in ChromaDB return Chroma.from_documents( documents = chunks, embedding = OpenAIEmbeddings(model="text-embedding-3-small"), collection_name = "book_rag", persist_directory= "./chroma_db", )
// code · step 2 · langgraph pipeline

Building the RAG Graph

rag_graph.py # ── 1. Shared State ─────────────────────────────────── class RAGState(TypedDict): query: str; retrieved_docs: List[str]; answer: str # ── 2. Node: retrieve ───────────────────────────────── def retrieve(state: RAGState) -> dict: docs = vectorstore.similarity_search(state["query"], k=4) return {"retrieved_docs": [d.page_content for d in docs]} # ── 3. Node: generate ───────────────────────────────── def generate(state: RAGState) -> dict: context = "\n\n---\n\n".join(state["retrieved_docs"]) resp = OpenAI().chat.completions.create(model="gpt-4o", messages=[ {"role":"system", "content":"Answer ONLY from context."}, {"role":"user", "content":f"CONTEXT:\n{context}\nQ: {state['query']}"}]) return {"answer": resp.choices[0].message.content} # ── 4. Build + compile ──────────────────────────────── graph = StateGraph(RAGState) graph.add_node("retrieve", retrieve) graph.add_node("generate", generate) graph.add_edge(START, "retrieve"); graph.add_edge("retrieve", "generate"); graph.add_edge("generate", END) app = graph.compile()
